# R1.1 发布基线调查报告

## 结论

- 正式版本建议为 `v1.14.0`，先固定 RC commit 再发布。`origin/master` 相对 `v1.13.4` 已有 89 个提交，包含新 `threading` 包、Python 3 语义修复、runtime/VM 优化和 fatal 错误可见性改进，不适合作为 `v1.13.5`。
- 现有发布由 `version_config.py`、`version.py`/`version.sh`、`push-core.sh`、`release_auto.py`、`packages.toml`、人工 tag 和人工 GitHub Release 串联；Actions 只覆盖 master push/PR，不响应 tag，也不创建 Release。
- `v1.13.4` 历史发布分成版本锁定、包索引更新、正式 release 三个提交，tag 为 lightweight tag；GitHub Release 存在，只有人工维护的说明和源码资产。

## 元数据与工具风险

- `src/PikaVersion.h` 和 `port/linux/version_config.py` 仍为 `1.13.4`，运行时编辑时间已漂移到 2025-06-02；CMSIS Pack 仍为 `1.13.0`。
- `release_helper.py` 把 patch 大于 9 自动进位到 minor，不符合 SemVer；`toml.dump` 会重排全表。
- `release_diff.py` 通过 checkout 历史提交和 master 生成差异，会扰动当前工作区。
- 没有统一检查 tag、运行时版本、版本配置、CMSIS Pack、包版本和包 commit 一致性的入口。

## 包范围

- `pikascript-core v1.14.0`、`PikaStdLib v1.14.0`。
- 新增 `threading v0.1.0`，当前不在 `packages.toml`。
- `requests v1.0.4`、`mqtt v0.1.2`、`time v0.2.3`，其源码均晚于当前索引 commit。
- `_thread`、STM32 等没有当前索引 commit 之后的有效源码变化，不机械升版。

## 发布阻断项

- `#372` optimize_speed runtime 缓存语义受分配布局影响，属于正式版 blocker。
- `#369` 合法现有子集误拒绝需要在完整 default 构建复验并收口。
- `#371` 当前默认 CI 排除了 REPL 不稳定测试，需修复测试合同并恢复覆盖。
- `#367`、`#368` 的修复提交看起来已进入 master，需要复验并关闭，避免 issue 状态漂移。
- `#373` 不阻断 runtime 正确性，但在解决前不能对单个性能提交宣称精确收益。
