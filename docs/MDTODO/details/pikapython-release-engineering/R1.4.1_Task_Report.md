# R1.4.1 任务报告

## 结论

已在现有 parser/ASM/VM 异常框架内修复 Issue #368 的两个剩余可局部闭环问题：handler 内再次抛出的异常可继续传播到外层 try；父类 handler 的 alias 按实际异常类型构造对象，不再错误绑定为父类实例。未新增 VM 指令，未修改 `PikaVMError` 布局，未扩大 typed except 语法范围。

## 实现

- 将线程现有 `try_state` 从二值用法改为同字段的嵌套深度计数；`TRY` 递增、`NTR` 递减，所有 try 内判断统一按非零处理。handler 内抛出时仍保留外层深度，因此继续跳转到外层 `NTR/EXP`。
- alias 字节码继续复用 `EXP -> JEZ -> OUT`。`EXP` 命中后根据错误栈中的 `exception_type` 复用既有异常哈希表，在 builtins 方法表定位实际异常构造器，构造对象并交给现有 `OUT`；parser 不再按 handler 声明类型执行 `RUN`。
- 为既有跨文件函数 `obj_getProp()` 补充头文件声明，消除隐式返回类型导致的指针截断风险。
- 本轮恢复的是 alias 的实际异常类型语义；原异常对象 identity、消息和参数未存入现有 `PikaVMError`，若要求完整保留需扩展错误状态模型，超出本轮非架构边界。

## 测试

- 失败基线：`except ArithmeticError as err` 中 `isinstance(err, ZeroDivisionError)` 为 0；内层 ValueError handler 再抛 TypeError 未被外层捕获。
- 修复后 typed-except runtime matrix 和 legacy untyped `EXP`：2/2 通过，每个场景析构后 `pikaMemNow()==0`。
- parser 合法/非法 typed except 邻接组合：5/5 通过。
- Docker 隔离网络、LVGL 和既定不稳定 REPL 项的最终 Linux 回归：455/455 通过。
- Valgrind 定向：50,586 alloc/free，退出时 0 bytes，0 errors。
- minimal 配置 Release core 完整编译通过，临时配置已恢复 default。
