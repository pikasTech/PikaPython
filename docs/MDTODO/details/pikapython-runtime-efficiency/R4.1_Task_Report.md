# R4.1 调查报告

## 结论

- issue #368 可以稳定复现：parser 会丢弃 `except` 类型和 alias。
- 原字节码中 bare、typed、alias 三种形式均生成无参数 `EXP`。
- 多个 handler 被顺序编译，异常发生时会依次执行全部 handler。
- VM 错误栈仅保存错误码，无法区分共用错误码的 `TypeError`、
  `ValueError` 和 `ZeroDivisionError`。
- `PikaVMError` 在指针前存在对齐空隙，可加入一个 `uint8_t` 异常类别而
  不改变结构体大小。

## 修复边界

- 保留 VM、对象模型和现有指令集。
- `EXP` 复用常量参数携带异常类型，并通过后续 `JEZ` 选择 handler。
- handler 链末复用 `GER` 和 `RIS` 传播未处理异常。
- alias 绑定为匹配 handler 对应的异常实例。
- tuple exception 和不完整 alias 不在现有子集内，稳定返回语法错误。
- handler 内再次抛出到外层需要重构线程级 try 状态，本轮不处理。

## 探针证据

- ASM 探针覆盖 bare、单 typed、多 typed、typed 加 bare 和 alias。
- 运行探针覆盖 25 个解析及运行场景。
- 修复前确认 7 个 Python 3 语义错误，另发现 `raise ValueError()` 被静默忽略。
