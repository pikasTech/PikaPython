# R1.7 内核设计评估

- 总体判断：
  - 不是“毫无设计的糟糕内核”，其超小型 MCU 目标非常明确。
  - 若按 Python 3 兼容性、健壮性、可维护性和执行效率评价，工程水平只能算中等偏下，技术债务明显。
- 合理设计：
  - `InstructUnit`、连续常量池、柔性数组 `Arg` 和序列化 VM 栈都在压缩 RAM/Flash。
  - 编译开关、minimal/optimize/pool/float/default 多配置和可替换平台层适合裸机移植。
  - 内存池与小对象缓存是有针对性的优化；测试中小对象缓存命中约 84%。
- 低效与脆弱点：
  - 每次函数调用都会分配 `argv` 和两个固定签名缓冲，并重新用字符串拆解参数签名。
  - 对象属性和局部变量存储是单链表线性查找，名称只保存 31 位 time33 哈希；碰撞没有原名二次校验。
  - 链表的移动缓存代码被固定关闭，`args_getSize` 也需要线性遍历。
  - 解析器大量依赖字符串重写和递归生成临时 AST；本轮 `#322` 的空值断言与段错误正是这种弱错误传播的表现。
  - `PikaVM.c`、`PikaObj.c`、`PikaParser.c` 分别约 5 千、5.2 千、3.9 千行，职责集中，回归风险高。
  - GC 在全局对象链上标记扫描，每轮最多释放 16 个对象；部分错误路径直接断言、abort 或永久循环。
- 性能证据：
  - NC01 Linux Debug 自带基准中，`prime_number_100` 的 Pika VM CPU 中位数约 9.49 ms，原生 C 显示约 0.002 ms。
  - 约数千倍差距被 Debug 模式和原生 C 常量优化显著放大，不能外推为 MCU 绝对倍率。
  - 结果仍说明该内核优化目标是体积和可移植性，不是吞吐量。
- 结论：
  - 对 4-64 KiB RAM 级设备，它是有价值且有针对性的实现。
  - 对需要较完整 Python 3 语义、稳定错误恢复或高性能的产品，它的当前架构不够好。
  - 本轮遵守约束，只做局部语法兼容和失败传播修复，不展开架构重构。
