# R14.12 任务报告

## 结果

- 修复 [Issue #370](https://github.com/pikasTech/PikaPython/issues/370)：普通 `threading.Lock` 不再直接依赖 owner-aware pthread mutex 的未定义双解锁行为。
- 普通 Lock 改为 mutex、condition 和 `locked` 状态组成的 ownerless 状态机，保留 Python 允许其他线程 release 普通 Lock 的语义。
- 对未锁定 Lock 调用 `release()` 现在抛出 `RuntimeError: release unlocked lock`；非法 RLock release 抛出 `RuntimeError: cannot release un-acquired lock`。
- 将原混合脚本拆为 `lock.py`、`lock_thread.py`、`rlock.py`，保留单个 `threading.lock_rlock` GTest 注册并在内部依次执行，恢复 TEST 与 VALGRIND 默认覆盖。
- 新增 `docker/Dockerfile.dev`，所有 Linux 构建、测试和 Valgrind 均在 Ubuntu 22.04 容器中通过 bind mount 执行。

## 稳定性处理

- `_thread` 的 VM 计数会先于 wrapper 资源清理归零；测试在 worker 场景下于主对象销毁后做有界 yield，等待异步清理完成并继续严格断言堆归零。
- 拆成三个独立 GTest 注册时，`optimize_speed` 下两个无关异常测试会随启动分配布局改变而失败；该独立 runtime 缓存债务已登记 [Issue #372](https://github.com/pikasTech/PikaPython/issues/372) 和 MDTODO R14.13，本轮采用单注册避免扩大范围。

## 验证

- Docker 五配置全量：minimal `217/217`、optimize_speed `450/450`、pool `442/442`、float `439/439`、default `451/451`。
- default 全量额外连续复测两轮，均为 `451/451`。
- Valgrind `threading.lock_rlock`：`41,429` 次分配与释放，退出时 `0 B`，`0 errors`，无泄漏。
- canonical 与 Linux vendored 的 `PikaPlatformEx.c/.h`、`threading_Lock.c`、`threading_RLock.c` 完全一致。

## 资源

- 路径归一化的 Release 静态库：`text` 从 `1,042,109 B` 增至 `1,042,597 B`，Flash 增量 `488 B`。
- `data` 保持 `126,480 B`，`bss` 保持 `24,342 B`，无全局常驻 RAM 增量。
- 每个实际创建的普通 Lock 平台状态从 `56 B` 增至 `96 B`，动态 RAM 增量 `40 B`。
