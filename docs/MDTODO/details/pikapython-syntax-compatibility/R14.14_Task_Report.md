# R14.14 任务报告

## 结论

本轮增加了一个预编译字节码的嵌入式控制循环复合 benchmark，并用该负载和仓库现有 gprof/Callgrind 工具定位 runtime 热点。最终保留的优化将 `strPointToLastToken()` 从最多三遍扫描改为单遍扫描，不增加分配、缓存或常驻状态，也不改变 VM/parser 架构。

## 代表性 benchmark

新增 `embedded_control_loop_1000`，覆盖对象构造、属性读写、方法调用、整数算术、条件分支、循环和控制器状态更新。parser 在计时区外预编译；每轮校验 `measurement == 12` 和 `checksum == 4790`，错误结果直接使 benchmark 失败。`performance/main.c` 同步改用该负载，重复 100 轮以供 gprof 采样，替换了原先不具代表性的 50 万次空 `print` 循环。

## 热点与优化

初始 gprof 中 `__pikaVM_runByteCodeFrameWithState` 约占 28.36%，`VM_instruction_handler_OPT` 约占 13.43%，`strPointToLastToken` 约占 4.48%。源码审计确认 `strPointToLastToken` 先调用 `strIsContain` 全量扫描，再调用 `strlen`，最后反向扫描，最坏读取同一字符串三遍。最终实现只前向扫描一次并保存最后一个分隔符后的位置。

曾验证两类未保留候选：错误栈检查缓存没有可信加速；整数 OPT 快路径虽减少 0.9% 至 3.6% 动态指令，但增加 288 至 912 B text；`OperatorInfo` 局部初始化增加 text 且触发 benchmark 结果错误。它们均在提交前退出。

## A/B 证据

固定 Docker 镜像、CPU 7、speed 配置和同一份预编译 workload，Callgrind 动态指令如下：

- `embedded_control_loop_1000`: 93,959,890 -> 93,545,227，减少 414,663（0.441%）。
- `while_loop_10000`: 63,646,549 -> 63,413,827，减少 232,722（0.366%）。
- `function_call_1000`: 66,629,036 -> 66,016,564，减少 612,472（0.919%）。
- 复合负载 Callgrind CPU 时间 735 ms -> 630 ms；该时间仅作辅助证据，主要判据是稳定的动态指令数。

宿主持续高负载导致原生 Google Benchmark CV 达 6.8% 至 14.3%，因此本轮不以墙钟数据宣称具体百分比加速；统计质量债务继续由 Issue #373 跟踪。

## 资源

- 默认 size 配置：text 1,567,491 -> 1,567,339，减少 152 B；data/bss 不变。
- benchmark speed 配置：text 1,475,394 -> 1,475,370，减少 24 B；data/bss 不变。
- 优化没有新增堆分配、缓存、全局字段或栈缓冲。

## 测试

- `dataString.point_to_last_token` 覆盖多级、无分隔符、前导、尾随和空字符串。
- 定向 GTest 3/3 通过。
- Valgrind 定向测试通过，5,510 次分配/释放，0 leak，0 error。
- LVGL 生成物隔离后，minimal、speed、pool、float、default 五配置回归完成；最终 default 轮 452/452 通过，脚本退出码 0。
- `git diff --check` 通过，网络类测试按既定过滤合同隔离。

## 踩坑记录

- `init.sh` 会删除 tracked 的 Linux LVGL 镜像并重写 PikaUI 生成资产；验证后必须精确恢复这些生成副作用，避免混入提交。
- Callgrind 在当前 Docker 环境必须使用 `--ulimit nofile=1024:1024`，否则 Valgrind 在 `vgPlain_safe_fd` 断言失败。
- 高负载宿主上的短时 benchmark 不能只看单次中位数；先用 gprof 定位，再用固定配置的动态指令数判断是否真实减少工作量。
