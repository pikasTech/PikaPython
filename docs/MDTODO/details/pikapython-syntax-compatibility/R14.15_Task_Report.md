# R14.15 Fibonacci runtime benchmark 与 VM 优化报告

## 交付

- 在 Google Benchmark 中增加 `fibonacci_recursive_20`，预编译 Python 源码后才进入计时循环，并校验结果为 `6765`。
- 将现有 performance profiling 入口切换为 10 轮 `fib(20)`，用于 gprof/Callgrind 热点采样，并校验结果。
- 增加递归全局函数查找回归测试，验证 `fib(10) == 55` 且退出后 `pikaMemNow() == 0`。
- 在 `VM_instruction_handler_RUN` 中避免递归函数调用对同一 `locals` 执行两次 `obj_getMethodArg_noalloc`；查找顺序与 VM 架构不变。
- 将仅在 `PikaVM.c` 内使用的 `PikaVMFrame_create/destroy` 设为 `static`，补偿优化带来的代码体积并释放编译器优化空间。

## 热点与根因

Release、benchmark speed 配置、带调试符号的 Callgrind 基线共执行 `337,485,343` 条指令。主要 inclusive 热点为 `_obj_getPropArg` 20.87%、`obj_getProp` 19.69%、`obj_getMethodArg_noalloc` 19.28%、`obj_deinit` 13.12%、`PikaVMFrame_loadArgsFromMethodArg` 10.77%、`VM_instruction_handler_OPT` 10.43%，并伴随约 21,891 个递归 Python 调用帧。

无限定名递归调用中，方法首次已在 `vm->locals` 查找；后续 fallback 又对同一对象重复查找，之后才检查 globals。此次优化只删除该重复查找，不改变 locals/globals 优先级、字节码格式或调用帧结构。

Debug gprof 的 10 轮 `fib(20)`耗时 `1.992444s`，分配 1,107,225 次，缓存命中 447,632 次（40.43%），热点同样集中在指令分派、方法/属性查找、参数复制和栈操作。Debug `-pg` 与 Callgrind 的 `SIGPROF` 冲突，因此动态指令结论使用无 `-pg` 的 Release Callgrind 数据。

## A/B 结果

- `fib(20)`：`337,485,343 -> 309,644,441`，减少 `27,840,902` 条指令，改善 8.25%。
- 控制项 `function_call_1000`：`66,009,892 -> 66,009,875`，变化 -17。
- 控制项 `while_loop_10000`：`63,429,087 -> 63,429,079`，变化 -8。
- core library：text `1,037,727 -> 1,037,375`，减少 352 B；data 保持 68,752 B；bss 保持 24,326 B。

曾验证直接把无点号 host 指向 locals 的方案，fib 改善 8.85%，但 text 增加 32 B，未采用。条件等价改写增加 48 B text，未采用。把 `ireg` 清零循环替换为 `pika_platform_memset` 虽减少 text，却使 fib 增加 459,730 条动态指令，按 runtime 优先原则未采用。

## 验证

- 定向 GTest `pikaMain.recursive_global_function_lookup` 通过。
- 定向 Valgrind：10,569 allocs / 10,569 frees，退出时 0 bytes，0 leaks，0 errors。
- Docker bind mount 下 `port/linux/gtest.sh` 五配置回归全部通过；各轮包括 219、452、441 和最终默认配置 453 项测试，网络类测试已隔离。
- 首轮受已跟踪 LVGL `.pyi` 与禁用实现不一致影响，按仓库 `init.sh` 隔离 LVGL 后通过；隔离产生的 tracked 删除和生成物均已恢复或清理。
